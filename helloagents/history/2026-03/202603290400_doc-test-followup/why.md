# 变更提案: MyBatis 迁移后的文档与测试补强收口

## 背景
当前 `JdbcTemplate -> MyBatis 注解 SQL` 迁移主线已经完成，后端业务代码中未再发现 `JdbcTemplate` 直接调用，核心认证链路与公告链路也已迁移到 Mapper / Provider。Codex 只读审查结论显示：

1. 代码层面未发现明显漏迁位置；
2. 仍存在少量**文档与方案包同步不足**；
3. 认证兼容字段分支和公告边界行为的**测试覆盖仍可补强**；
4. 暂未发现需要变更业务逻辑的高优先级缺陷。

因此，本轮后续收口应限定在：
- **补测试**，验证兼容分支与公告边界；
- **补文档**，让方案包/知识库与最终代码状态一致；
- **不改业务逻辑**，避免在迁移已稳定后再引入行为漂移。

## 目标
1. 补齐认证兼容字段的测试矩阵，至少覆盖历史列名与默认列名两条路径；
2. 补齐 Bulletin 的空 `expiredAt`、创建人持久化、列表排序/格式等测试；
3. 修正文档与方案包中关于迁移范围、月度归档、最终落地状态的描述；
4. 保持业务代码行为不变，只通过测试与文档提升可追溯性。

## 非目标
- 不调整认证/公告业务规则；
- 不新增权限判断逻辑；
- 不继续做新的 Mapper 迁移；
- 不做性能优化（如 `information_schema` 列探测缓存化）。

## 成功标准
- 新增/调整的测试能稳定通过；
- `history/index.md`、`CHANGELOG.md`、`wiki/arch.md`、迁移方案包与代码现状一致；
- 无业务接口返回、权限行为、日志行为变化。
