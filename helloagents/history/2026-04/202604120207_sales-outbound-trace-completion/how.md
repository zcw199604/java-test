# 技术方案：销售出库选仓闭环 + 库存统一追溯模型

## 1. 方案概览
本次采用“最小侵入增强”的路径，在保留现有 `trace_records` 与 `inventory_records` 双表结构的前提下，补齐销售出库前端交互，并通过后端聚合查询将库存动作收口为统一追溯视图，而不是直接重构现有数据模型。

## 2. 推荐方案
### 方案A（推荐）：保留双记录表，新增统一库存追溯查询层
- 前端新增独立销售出库页 `SaleOutboundView.vue`，复用现有销售详情与仓库接口。
- 后端继续保留：
  - `trace_records`：记录业务节点；
  - `inventory_records`：记录库存变化。
- 在库存动作执行时补充统一 trace 写入，确保采购入库、销售出库、调拨、盘点都能进入 `trace_records`。
- 新增统一追溯查询接口，对两类数据进行聚合或分层输出，供“库存追溯台账 / 合规追溯”页面消费。

#### 优点
- 对现有表结构改动最小，迁移成本低。
- 能兼容当前采购 / 销售追溯实现，风险较低。
- 前后端改造范围清晰，适合当前项目规模。

#### 缺点
- 追溯聚合逻辑会略复杂。
- 前端需区分“业务节点视图”和“库存变化视图”的展示层次。

### 方案B：将所有库存与业务节点完全并表为新追溯模型
#### 优点
- 语义统一，理论上更整洁。
#### 缺点
- 需要迁移历史写入逻辑、接口、测试与文档，风险明显更高。
- 对当前项目而言属于过度设计。

## 3. 关键设计
### 3.1 销售出库页面补齐
- 新增路由 `/sale/outbound`。
- 新页面需支持：
  - 读取订单 ID；
  - 拉取销售订单详情；
  - 拉取可用仓库列表；
  - 用户选择仓库并填写备注；
  - 调用 `POST /api/sales/{id}/outbound`。
- 订单列表页“出库”按钮保持跳转模式，与采购入库页交互一致。

### 3.2 库存统一追溯模型
对以下动作统一补充 `trace_records` 写入：
- 采购入库：已存在，保留；
- 销售出库：已存在，保留；
- 调拨：新增 `TRANSFER_OUT` / `TRANSFER_IN` 或统一 `TRANSFER` 节点说明；
- 盘点：新增 `CHECK` 节点。

查询层建议提供两种视角：
1. **业务节点视角**：强调审核、到货、入库、出库、盘点、调拨等节点事件；
2. **库存变化视角**：强调 beforeQty / afterQty、仓库变化、数量变化。

### 3.3 查询接口增强
优先在现有报告或库存模块中新增统一查询接口，支持以下筛选维度：
- 订单号
- 商品 ID / 商品名称
- 仓库 ID
- 业务类型（PURCHASE / SALES / INVENTORY）
- 节点类型（INBOUND / OUTBOUND / TRANSFER / CHECK）

### 3.4 前端追溯展示增强
- `InventoryLedgerView.vue`：增强筛选能力，可查看统一库存链路。
- `TraceQueryView.vue`：从“合规追溯列表”升级为“统一业务 / 库存追溯查询”入口，支持更精确筛选与分组展示。

## 4. 数据与兼容性
- 优先不新增数据库表。
- 如需补字段，应仅做兼容性追加，避免破坏历史初始化脚本。
- 历史 `trace_records` 数据继续可读，新逻辑只追加节点，不回写旧数据。

## 5. 测试策略
### 阻断性测试
- 销售出库接口：仓库必填、库存不足、成功出库。
- 统一追溯接口：能查到采购入库 / 销售出库 / 调拨 / 盘点节点。

### 警告性测试
- 前端页面路由跳转、提交成功后的回跳与提示。
- 追溯页筛选结果与展示顺序。

## 6. 风险与规避
- 风险1：追溯语义不统一
  - 规避：统一定义 nodeCode / nodeName 命名规则，并在文档中记录。
- 风险2：前端新页面与现有权限模型不一致
  - 规避：复用 `order:approve` / `inventory:edit` 现有权限约束。
- 风险3：追溯查询性能下降
  - 规避：先以筛选查询 + 有限结果集实现，避免一次性全量聚合。

## 7. ADR
- ADR-20260412-01：采用“保留 `trace_records` + `inventory_records` 双模型，通过统一查询层收口”的方案。
