# 技术方案: 文档与实现对齐及报表/采购流程修复

## 方案概述
采用“最小必要修复”的方式，同时修改后端接口语义、前端消费逻辑和知识库文档，避免额外架构重构。所有变更围绕“以当前后端真实能力为准”展开：
- 报表导出接口改为 CSV 文本输出；
- 报表中心使用库存台账接口展示明细；
- 采购流程引入独立到货状态；
- 首页统计改为数据库聚合值；
- 文档与知识库同步更新。

## 关键设计

### 1. 报表导出
- 保持路径 `/api/reports/export` 不变。
- 后端返回 `text/csv; charset=UTF-8`，内容为汇总 + 趋势的 CSV 文本。
- 前端预览使用文本读取，下载使用 Blob 下载，二者共享同一接口。

### 2. 报表中心前端联调
- `inventory-summary` 仅用于汇总卡片，不再误用作库存明细表。
- 库存明细表改为调用 `/api/inventories`。
- 商品列表的品类列改用后端真实字段 `category`。

### 3. 采购状态流转
- 状态定义：`CREATED`（已创建）→ `RECEIVED`（已到货）→ `INBOUND`（已入库）。
- 增加 `received_at` 字段记录到货时间。
- `receive` 仅负责到货登记；`inbound` 仅允许对已到货采购单执行入库并更新库存流水。

### 4. 首页概览
- 采购单、库存记录、销售单、预警数量均改为数据库实时查询。
- 快捷模块仍由后端统一返回，保持前端布局不变。

## 风险与规避
- **数据库兼容性风险:** 现有表可能已存在。通过在 schema 中补充基于 `INFORMATION_SCHEMA` 的条件 DDL，保证已建表场景也可升级。
- **前端下载兼容性风险:** 导出接口改为 CSV 后，预览与下载需统一改造，避免继续将 JSON 当 CSV 处理。
- **状态流转风险:** 入库前强制要求已到货，避免采购单重复入库或跳步流转。
