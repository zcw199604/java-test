# 变更提案：补齐销售出库选仓页面与库存全链路统一追溯

## 1. 背景
当前版本中，销售出库后端接口已支持按仓库出库、库存扣减与库存流水写入，但前端订单页仍跳转到未注册的 `/sale/outbound` 路由，导致“库管选择仓库后执行出库”的用户闭环未完全落地。

同时，系统已具备两类追溯能力：
- 采购 / 销售业务节点追溯：通过 `trace_records` 记录审核、到货、入库、出库、回款等动作；
- 库存台账追溯：通过 `inventory_records` 记录入库、出库、调拨、盘点等库存变化。

但二者仍然分裂，尚未形成覆盖采购、调拨、盘点、销售的统一库存全链路追溯模型，导致用户需要在多个页面之间切换，无法按“商品 / 订单 / 仓库 / 业务链路”一次性查看全流程。

## 2. 问题陈述
### 2.1 销售出库前端闭环缺口
- 订单列表页存在“出库”动作，但路由目标 `/sale/outbound` 未注册。
- 用户无法在正式页面中选择仓库并提交出库。
- 现有 README / 知识库口径与真实前端交互存在差距。

### 2.2 库存全链路追溯能力缺口
- `trace_records` 主要覆盖采购 / 销售节点，库存模块自身未统一写入 trace。
- `inventory_records` 虽包含调拨、盘点、入出库明细，但没有与统一业务追溯模型收口。
- 合规追溯页目前以通用列表筛选为主，尚未提供“库存主线”维度的统一展示。

## 3. 目标
1. 补齐销售出库独立选仓页面或等效交互闭环，使库管/管理员可从前端完成“选择仓库 → 出库”。
2. 将库存相关关键动作（采购入库、销售出库、仓间调拨、库存盘点）统一纳入可查询的追溯模型。
3. 提供面向用户的一致化追溯查询入口，支持从库存视角查看业务链路。
4. 同步更新知识库与文档口径，确保“代码即事实”。

## 4. 范围
### 范围内
- 销售出库前端页面、路由与提交流程补齐
- 后端库存相关 trace 模型扩展与查询接口增强
- 统一追溯页面 / 库存追溯台账交互增强
- 相关测试、文档与知识库更新

### 范围外
- 销售部分出库、批次拆分、撤销出库
- 调拨审批流、调拨单据状态机
- 复杂仓储批次管理、条码 / RFID 能力

## 5. 成功标准
- 前端存在可访问的销售出库选仓页面或同等级别稳定交互，用户能完成出库操作。
- 采购入库、销售出库、调拨、盘点四类库存动作均能进入统一追溯链路。
- 库存追溯页面可按关键条件查询并展示完整链路。
- 自动化测试覆盖新页面交互依赖的后端接口与统一追溯核心查询。
