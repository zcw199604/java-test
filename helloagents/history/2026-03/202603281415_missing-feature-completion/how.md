# 技术方案: 缺失功能补齐与消息中心完善

## 方案概述

采用“最小侵入扩展”方案：在尽量复用现有后端服务与前端页面结构的前提下，补齐缺失接口、完善现有路由与页面交互，避免大规模重构已有权限模型和数据库结构。

## 方案选择

### 方案1（最小侵入修复-推荐）
- 复用现有 AuthController / SystemController / MessageController / ReportController
- 为订单补充 detail + update 能力
- 为客户、仓库补前端页面并适配现有/新增接口
- 在现有报表页与库存页增量补齐功能
- 新增独立消息中心页并复用现有消息接口

**优点:** 风险低、与现有代码风格一致、便于快速验证
**缺点:** 仍保留部分历史结构限制，如基础资料模块页面分散

### 方案2（模块重构）
- 重组基础资料、消息、统计页结构
- 统一权限命名与路由组织

**优点:** 长期维护性更高
**缺点:** 影响范围大，不适合本次快速补齐任务

## 技术设计

### 1. 认证前端闭环
- 登录页调用 `/api/auth/captcha`
- 新增忘记密码对话框，调用 `/api/auth/forgot-password`
- 新增重置密码页面，调用 `/api/auth/reset-password`
- 个人中心改密调用 `/api/profile/password`

### 2. 基础资料管理完善
- 客户：新增独立管理页面，支持列表、创建、编辑、停用
- 仓库：新增独立管理页面；后端新增 `GET /warehouses/{id}`、`PUT /warehouses/{id}`、`PUT /warehouses/{id}/status`
- 在基础资料概览页保留总览职责，不承担完整编辑职责

### 3. 订单编辑实现
- 采购：新增 `GET /api/purchases/{id}`、`PUT /api/purchases/{id}`
- 销售：新增 `GET /api/sales/{id}`、`PUT /api/sales/{id}`
- 状态限制：仅 `CREATED` / `REJECTED` 可编辑，避免影响已审核单据

### 4. 库存功能完善
- 在库存总览页增加预警摘要与状态筛选
- 保持现有库存导入、流水、盘点和调拨主逻辑不变
- 与仓库管理页面联动，提升库存场景闭环

### 5. 统计与追溯完善
- 报表大屏接入 `/reports/psi-summary` 与 `/reports/linkage`
- 新增采销存联动图表、应收账款等辅助指标展示
- 不额外新增后端复杂聚合接口，优先复用已有接口

### 6. 消息中心
- 新增 `/message/center` 页面
- 复用 `/api/messages` 和 `/api/messages/{id}/read`
- 前端支持关键字、已读状态筛选、全部已读

## 安全与性能

- 所有新增更新接口延续参数化 SQL，避免拼接 SQL
- 订单更新限制可编辑状态，防止已完成流程被篡改
- 密码相关表单增加前端校验，但真实安全判断以后端为准
- 消息中心仅展示当前登录用户和公共消息

## 架构决策 ADR

### ADR-20260328-feature-gap-closure
- **决策:** 采用增量补齐方案而非大规模重构
- **原因:** 当前仓库已有主要业务骨架，本次目标是快速与功能清单对齐
- **影响:** 改动分布在多个模块，但单点逻辑清晰，可控可验
