# 技术方案: JdbcTemplate 向 MyBatis 注解 SQL 渐进迁移

## 方案概述

本次采用 **测试先行 + 分阶段迁移 + 注解 SQL 优先** 的方案。

核心思路：

1. 先把现有关键数据库交互行为通过测试“钉住”；
2. 再引入 MyBatis 基础设施，但不立即全量替换；
3. 按“低风险模块 → 中风险模块 → 高风险模块”逐步迁移；
4. 每完成一批 Mapper 替换，立即执行全量回归；
5. 对极少数不适合注解表达的复杂 SQL，设置严格例外准入标准。

---

## 方案选择

### 方案1（测试先行 + 渐进式注解迁移 - 推荐）

**做法:**
- 新增 MyBatis 依赖与配置；
- 保留现有 JdbcTemplate 代码作为过渡；
- 先补测试，再分模块迁移；
- 新增 `mapper` 包，优先使用 `@Select / @Insert / @Update / @Delete / @Results / @ResultMap / @SelectProvider`；
- 每迁移一个模块就做完整回归。

**优点:**
- 风险最可控；
- 可逐步观察行为一致性；
- 适合当前多模块、多兼容逻辑的代码现实；
- 出现问题时容易定位到最近一批迁移。

**缺点:**
- 迁移周期较长；
- 过渡期会短暂同时存在 JdbcTemplate 与 Mapper 两套访问风格。

### 方案2（一次性全量替换为 MyBatis 注解）

**做法:**
- 统一新增 Mapper 后，直接替换全部 `JdbcTemplate` 调用。

**优点:**
- 代码风格切换快；
- 理论上能更快达成最终目标。

**缺点:**
- 风险极高；
- 出问题时难以定位；
- 与“先补测试、保证不引入新问题”的要求冲突。

### 方案3（MyBatis + XML Mapper 为主）

**做法:**
- 大量 SQL 迁移到 XML 中，用接口绑定。

**优点:**
- 复杂 SQL 表达力强；
- 动态 SQL 管理集中。

**缺点:**
- 与用户“优先使用注解方式写 SQL”的要求不一致；
- 需要额外维护 XML 资源与映射关系。

### 推荐结论

采用 **方案1**。

原因：
- 与用户要求完全一致；
- 与当前项目复杂度匹配；
- 能最大程度降低迁移引入新问题的概率。

## 迁移原则

### P1. 测试先行
- 不允许在核心数据库访问尚未建立测试基线前大规模替换；
- 先写失败/边界/兼容行为测试，再开始 Mapper 化；
- 新增测试必须覆盖“正向 + 反向 + 边界 + 兼容”。

### P2. 注解 SQL 优先
优先采用以下方式：
- `@Mapper`
- `@Select`
- `@Insert`
- `@Update`
- `@Delete`
- `@Results`
- `@Result`
- `@Options`
- `@SelectProvider / @UpdateProvider / @InsertProvider / @DeleteProvider`

### P3. XML 作为受控例外
仅当满足以下任一条件时，才允许引入 XML Mapper：
- 单条 SQL 的动态分支过多，注解可读性明显恶化；
- 结果映射层级复杂，注解维护成本过高；
- Provider 方案会显著降低可维护性。

⚠️ 不确定因素: 当前 `AuthService` 中存在较多“字段是否存在”的兼容逻辑，这类逻辑可能需要 `Provider` 或辅助基础设施支持。基于当前信息，仍优先按“注解 + Provider”设计；若落地时发现可读性过差，可将其列为 XML 例外候选。

### P4. 行为保持不变
- API 输入输出结构不变；
- 权限校验逻辑不变；
- 审计日志、登录日志、追溯、消息通知行为不变；
- 金额和数量计算精度不变；
- 兼容字段逻辑不变。

### P5. 分层清晰化
目标结构建议：

```text
com.example.tobacco
├─ auth
├─ audit
├─ ...
├─ mapper
│  ├─ auth
│  ├─ audit
│  ├─ system
│  ├─ purchase
│  ├─ sales
│  ├─ inventory
│  └─ ...
├─ model / dto
└─ service
```

## 测试优先设计

## 1. 测试总体策略

迁移前先把测试分为三层：

### A. 平台级集成测试
基于现有：
- `PlatformIntegrationTest`
- `BackendCompatibilityIntegrationTest`

扩展覆盖：
- 登录 / 登出是否正确写入 `login_logs` 与 `operation_logs`
- 兼容字段仍可工作（如 `uuid/code/token` 兼容）
- 仓库 / 客户 / 采购 / 销售更新链路完整通过

### B. 模块级集成测试
建议新增：
- `AuthPersistenceIntegrationTest`
- `SystemModuleIntegrationTest`
- `PurchaseModuleIntegrationTest`
- `SalesModuleIntegrationTest`
- `InventoryModuleIntegrationTest`
- `AuditAndMessageIntegrationTest`
- `ReportModuleIntegrationTest`

### C. Mapper 行为测试 / Provider 测试
对于复杂 Provider 逻辑，建议增加：
- SQL 构建分支测试
- 兼容字段分支测试
- 结果映射测试

## 分阶段实施设计

## 阶段 0：引入 MyBatis 基础设施（不替换业务）

### 目标
- 增加 MyBatis 依赖与自动扫描
- 增加基础配置
- 验证与现有 Spring Boot / JdbcTemplate 并存运行

### 设计点
- 保留 `JdbcTemplate`，先不删旧代码；
- 配置 `@MapperScan`；
- 确定统一包路径与命名规范；
- 增加一个最小试点 Mapper 做冒烟验证。

### 验证
- `mvn test`
- `mvn -DskipTests compile`
- 试点接口冒烟通过

## 阶段 1：先测试补齐

### 目标
- 把关键行为基线补齐
- 保证迁移前已有“红线”保护

### 输出
- 新增/扩展集成测试类
- 明确每个模块迁移的测试进入条件

### 进入下一阶段条件
- 核心链路测试通过
- 能稳定复现实有行为

## 阶段 2：低风险模块迁移

### 建议顺序
1. `message`
2. `audit`
3. `supplier`
4. `customer`
5. `catalog`

### 原因
- SQL 相对简单；
- 状态机少；
- 可快速验证 Mapper 组织方式是否合理。

## 阶段 3：中风险模块迁移

### 建议顺序
1. `system`
2. `report`
3. `dashboard`

### 风险点
- 动态筛选和聚合查询增多；
- 仓库 `code` 兼容逻辑；
- 用户、角色、权限多表操作。

## 阶段 4：高风险模块迁移

### 建议顺序
1. `purchase`
2. `sales`
3. `inventory`

### 风险点
- 状态流转复杂；
- 金额、库存数量、消息通知、审计日志相互联动；
- 事务边界要求高。

## 阶段 5：认证与安全模块迁移

### 建议顺序
1. `auth/AuthService`
2. `auth/ShiroRealm`
3. `interceptor` 相关查询支撑

### 风险点
- 登录、验证码、会话、密码重置均存在兼容字段逻辑；
- 一旦迁移失误，平台整体不可登录。

因此认证模块放在最后，待 Mapper 规范和测试体系成熟后再替换。

## 注解 SQL 规范

## 1. Mapper 命名
- `XxxMapper`：面向单表或聚合读写职责
- `XxxQueryMapper`：复杂查询可拆分
- `XxxSqlProvider`：复杂动态 SQL 生成器

## 2. 方法设计
- 单一职责：一个 Mapper 方法只承载一个明确查询/更新动作
- 参数显式命名：使用 `@Param`
- 返回值收敛：优先返回明确 DTO / Item，而非大量 `Map<String,Object>`

## 3. 结果映射
- 简单字段可依赖驼峰映射；
- 关键对象统一使用 `@Results` 或共享 `@ResultMap`；
- 金额类字段统一保证 `BigDecimal` 精度不漂移。

## 4. 动态 SQL
优先级：
1. 简单条件拼接 → 注解直接写；
2. 中等复杂度动态条件 → `Provider`；
3. 超复杂且注解/Provider 已明显不可维护 → 进入 XML 例外评审。

## 5. 事务策略
- 事务仍由 Service 层控制；
- 不把事务下沉到 Mapper；
- Mapper 保持数据访问职责单纯。

## 风险评估与规避

### 风险1：测试不足导致迁移后行为漂移
**规避:** 先补测试，未达基线不允许替换。

### 风险2：注解 SQL 可读性下降
**规避:** 提前定义 Provider / XML 例外边界；严格控制单方法复杂度。

### 风险3：兼容字段逻辑被破坏
**规避:** 认证模块最后迁移；为兼容字段增加专门测试。

### 风险4：金额/时间/编码序列化差异
**规避:** 对金额精度、时间格式、UTF-8 编码继续保留回归断言。

### 风险5：事务拆分不当
**规避:** Mapper 仅替换数据访问，事务边界不变，先以等价迁移为原则。

### 风险6：大批量改造导致定位困难
**规避:** 采用小批次迁移，每个阶段可单独回归与回滚。

---

## 安全与性能

- 继续坚持参数化 SQL，避免字符串拼接带来的注入风险；
- 审计日志、登录日志、追溯记录不可因迁移丢失；
- 不在迁移过程中修改权限模型与接口鉴权策略；
- 对查询性能敏感的报表 / 列表接口，在 Mapper 化后复核 SQL 执行计划；
- 对批量导入类场景，避免因 Mapper 改造引入 N+1 或重复查询问题。

## 架构决策 ADR

### ADR-20260329-mybatis-annotation-migration
- **决策:** 采用“测试先行 + 渐进式 MyBatis 注解迁移”方案。
- **原因:** 当前 JdbcTemplate 使用范围广，认证兼容逻辑与业务状态机复杂，一次性全量替换风险过高。
- **影响:** 项目将短期进入 JdbcTemplate 与 MyBatis 并存阶段，但可显著降低迁移风险，并为后续统一数据访问层奠定基础。
